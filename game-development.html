I created a basic WASD movement script for a spaceship in Unity, but it felt clunky and not very “spaceship-like”. To improve it, I contacted Granite via Ollama with the prompt:

“I have a Unity C# spaceship movement script. Can you tweak it to feel more ‘spaceshipy’, maybe with acceleration and inertia?”

Granite suggested applying velocity gradually instead of setting it directly, introducing acceleration and inertia for smoother, more realistic movement. I then refined this further by clamping speed and reducing jitter.

Once the movement felt stable, I synced it with a peer’s fuel meter animation. I asked Granite whether a GIF or sprite sheet would be better in Unity. Although it didn’t correctly identify GIF limitations in Unity, it still provided useful general comparisons between formats.

I then tested more specific Unity workflows, asking how to split sprite sheets and convert them into animations. This time Granite gave more accurate Unity-related guidance, though it was still somewhat general and didn’t fully address every part of the question.

From these experiments, I found Granite works best when given clear, focused prompts and higher-level problems rather than highly specific engine-edge cases. It is particularly useful for structured explanations and development guidance, but less reliable for precise implementation details.

I also tested Granite in a RAG setup using a Unity-based demo project. It performed well when working with limited, structured context, effectively staying within the provided information and delivering concise responses. After modifying the system prompt to enforce strict in-context answering and a sci-fi assistant persona, it followed instructions consistently and reliably.

Finally, I created a gameInstructions.txt file defining gameplay rules, objectives, scoring, and mechanics. Granite responded well to these instructions and accurately conveyed the game logic in-character as a spaceship AI assistant.
